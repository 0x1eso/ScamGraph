개요 · 비전 · 핵심 원칙

한 줄 정의와 문제의식

ScamGraph는 의심스러운 URL · 전화번호 · 계좌번호를 넣으면 규칙 엔진과 공개 데이터로 위험도와 그 근거를 즉시 설명하고, 수집된 사기 인프라(도메인·IP·전화·계좌)를 Neo4j 관계망으로 이어 "사기 조직"을 시각화하는 실시간 위협 인텔리전스 웹 플랫폼이다.

메이커톤 주제 "디지털 시민으로서 안전한 디지털 세상"(SDG 16 평화·정의·제도, SDG 10 불평등 완화 — 정보 취약계층 보호)에 대한 출품작이다. 보이스피싱·스미싱·피싱 사이트는 여러 채널(웹·문자·전화)에서 동시에, 조직적으로 이뤄진다. 단일 URL 검사기로는 "이 링크가 위험하다"까지만 말하지만, ScamGraph는 같은 IP·인증서·전화·계좌를 공유하는 캠페인 전체를 드러내고, 그 판정을 브라우저 확장과 모바일 앱으로 배포해 다음 피해자를 선제적으로 막는다.

4대 불변 원칙

순수 SW · AI 금지

모든 위험 판정은 명시적 규칙으로만 이뤄진다. 블랙박스 신경망이 없다. "왜 위험한가"를 언제나 재현·설명할 수 있는 것이 세일즈 포인트다.

설명 가능성(reasons)

모든 판정에 {rule, weight, detail, confidence?} 근거 배열이 붙는다. 점수만 단독 노출하지 않는다. UI·PDF·알림 어디서든 근거가 함께 간다.

데모 세이프

외부 네트워크·DB·브로커가 모두 죽어도 API와 화면은 시드/폴백으로 즉시 응답한다. quick_assess는 절대 네트워크를 호출하지 않는다.

프론트는 게이트웨이만 호출

웹은 엔진을 직접 부르지 않는다. 모든 접점(웹·확장·모바일)은 게이트웨이의 통합 엔드포인트 하나만 알면 된다 = "공용 두뇌, 얇은 클라이언트".

무엇이 다른가(차별점)

ScamGraph는 "피싱 URL 검사기"가 아니라 네 가지를 동시에 보여준다:

  • ① 실시간 다중채널 예방 — 웹 스캔 + 브라우저 확장(DNR 로컬 차단) + 모바일(전화·문자 스크리닝)이 하나의 판정을 공유.
  • ② 범죄 인프라 상관분석 — 공유 IP/인증서/등록자로 흩어진 도메인을 하나의 캠페인으로 묶는 그래프 귀속.
  • ③ 시민 신고 공유 방어망 — 커뮤니티 신고가 쌓이면 전 사용자가 보호되는 플라이휠(단, poisoning 방어 필수).
  • ④ 블랙박스 AI 없는 설명가능 보안 제품 — 모든 판정이 규칙·증거로 재현되고, 오탐은 이의제기·정정 피드로 회수된다.

계약 기반 병렬 개발

세 파트(엔진 / 게이트웨이+웹 / 인프라)의 유일한 접점은 API 계약이다. 응답 JSON 스키마(특히 /graph의 nodes/edges, grade ∈ safe|caution|warning|danger)만 지키면 서로 기다리지 않고 병렬로 개발할 수 있다. 스키마를 바꾸면 상호 협의가 원칙이다. 실제로 이 프로젝트는 다수의 트랙(엔진 정확도·피드·그래프분석·PDF·알림·신고방어·웹·확장·모바일·인프라·법률)으로 병렬 실행됐다.

컴포넌트별 "어떻게 동작하는가"

이 장이 이 문서의 핵심이다. 각 부분이 구체적으로 어떤 메커니즘으로 동작하는지 — 신호 하나하나의 가중치, 폴백 경로, 데이터 모양까지 — 설명한다.

규칙 엔진 quick_assess — 무네트워크 즉시 판정

파일: apps/engine/app/crawler.py. 입력(URL/전화/계좌)을 받아 네트워크를 전혀 건드리지 않고 위험 점수(0–100)와 근거 배열을 즉시 반환한다. 데모 세이프의 심장이다. tldextract는 suffix_list_urls=() · cache_dir=None으로 생성해 패키지 번들 스냅샷만 쓰고 네트워크/디스크 접근을 원천 차단한다.

처리 순서

  1. 입력 분류 classify_target(): 영문자/.// 포함 → url; 숫자 9–11자리 → phone; 10–16자리 → account.
  2. 화이트리스트 즉시 안전: 호스트가 ALLOWLIST(약 40개 정상 도메인, 하위 도메인 포함)면 즉시 safe 0점 반환. 반드시 원본 호스트로 검사한다 — 키릴 위장 nаver.com은 스켈레톤이 naver.com이라도 통과하면 안 되기 때문.
  3. 혼동문자 접기: decode_idna()로 퓨니코드를 유니코드로 되돌리고 confusable_skeleton()으로 ASCII 스켈레톤을 만들어 토큰·브랜드 분석을 견고하게 한다.
  4. eTLD+1 분해: tldextract로 co.kr·go.kr 같은 멀티파트 TLD까지 정확히 분해(_registrable). 실패 시 labels[-2] 근사로 폴백 → 절대 예외를 던지지 않는다.
  5. 규칙 누적 채점: 아래 신호들을 순차 적용해 점수를 더하고 각 reason을 남긴다. 최종 min(score,100).

왜 이게 중요한가 이 함수는 네트워크·DB·브로커가 전부 죽어도 밀리초 단위로 근거와 함께 결과를 낸다. 심사장 와이파이가 끊겨도 킬샷은 항상 동작한다. "AI 없이 규칙만으로도 이만큼 정확하고 설명 가능하다"가 핵심 메시지다.

심층 수집 crawl_and_enrich + score_enrichment

quick_assess가 즉답을 낸 뒤, Celery 워커가 best-effort로 실제 네트워크 수집을 수행한다. 모든 호출은 try/except + 타임아웃으로 감싸 실패해도 부분 결과를 반환한다(예외 없음).

SSRF·데모 세이프 심층 수집은 워커에서만, 사용자가 명시 실행한 스캔에 한해 동작한다. 크롤러는 private IP·metadata endpoint 차단 등 SSRF 방어가 로드맵상 필수 요건이다. quick_assess 경로에는 어떤 네트워크 호출도 없다.

게이트웨이 — 통합 두뇌

게이트웨이(apps/gateway, Java 21 · Spring Boot, 가상 스레드)는 모든 접점의 단일 접점이자 "공용 두뇌"다. 핵심은 통합 판정 GET /api/check이다.

통합 판정 GET /api/check?value=

브라우저 확장·PWA·모바일 앱이 이 엔드포인트 하나만 알면 되도록, 규칙 판정(엔진) + 조직 귀속(그래프) + 커뮤니티 신고 + 외부 피드 대조 + 실행 권고를 한 번에 합성한다. 각 단계는 실패해도 다음으로 넘어가는(try/catch) 데모 세이프 구조다.

단계동작등급 영향
0) 입력 검증길이 > 2048자 → grade:"unknown" 즉시 반환방어
1) 규칙 판정엔진 POST /scan 호출 → kind·grade·risk_score·reasons기본 등급
2) 조직 귀속GraphSource BFS로 최근접 Campaign 탐색연루 시 → danger
2b) 커뮤니티 신고SELECT COUNT(*) FROM reports WHERE target=?≥3 → danger, ≥1 → warning
2c) 외부 피드 대조blocklist 조회(value 또는 host) → external_feed_hit(weight 40, source·first_seen)등재 시 → danger
3) 실행 권고kind·grade·org별 문구 + 커뮤니티/피드 접두—
 
GET /api/check?value=shinhan-otp.xyz →
{ "value":"shinhan-otp.xyz", "kind":"url", "grade":"danger", "risk_score":73,
  "reasons":[ ...엔진 근거..., {"rule":"external_feed_hit","weight":40,"source":"openphish",...} ],

설명 가능성 유지 외부 피드 대조도 "어느 피드에 언제 등재됐는지"를 reason으로 남긴다. 게이트웨이는 판정을 합성하되 근거를 버리지 않는다.

신고 poisoning 방어 — ReportDefense

커뮤니티 플라이휠은 강력하지만 동시에 공격 표면이다. 한 행위자가 무고한 도메인을 대량 허위신고해 danger로 승격시키면 업무방해·괴롭힘이 된다. 오염 지점은 /api/check가 COUNT(*) FROM reports로 신고 수를 읽어 등급을 올리는 곳이므로, 접수 단계에서 게이팅해 reports에는 "dedup을 통과한 독립 신고자의 첫 기여"만 쌓이게 한다 → 원시 행수 = 독립 신고자 수 → "한 명이 아무리 반복해도 1".

계층메커니즘상수
신고자 익명 해시헤더(X-Forwarded-First-IP / X-Client-Id / XFF / remoteAddr)에서 식별자를 뽑아 pepper + SHA-256. 원시 IP는 저장 안 함REPORT_HASH_SALT
Dedup동일 신고자·동일 대상은 전 기간 1회만 집계—
Rate limit(burst)시간창 내 독립 신고 상한 초과 시 throttle창 3600s / 상한 20
독립 신고자 카운트등급을 좌우하는 수 = COUNT(DISTINCT reporter_hash)다수 신고 임계 3
Allowlist 가드정상 도메인(엔진과 동일 집합) 신고는 status='review' 검토 큐로만, 자동 승격 금지—

승격 정책: ① 1건=저장만 → ② 독립 다수(≥3)="다수 신고" → ③④⑤ 위험후보·확인·전체 blocklist는 강한 기술신호 결합 + 운영자 dual-approval이 있어야 하며 신고 접수 경로에서 자동 승격하지 않는다(blockEscalationEligible 훅). DB가 죽어도 정상 도메인만은 자동 승격을 막는 폴백을 낸다.

WebSocket 실시간 피드 허브 — /ws/feed

FeedHandler(TextWebSocketHandler)가 접속한 모든 세션에 스캔·신고 이벤트를 브로드캐스트한다. 실이벤트가 없어도 4초마다 시드 더미 이벤트를 순환(@Scheduled(fixedRate=4000))해 피드가 항상 "살아있게" 한다(데모 세이프). 다른 빈이 broadcast()로 실이벤트를 주입한다. 죽은 세션 하나가 루프 전체를 깨지 않도록 격리한다.

90초 데모 스크립트 — "한 흐름으로 전부 전달"

탐지·그래프·실시간·멀티플랫폼·사회적 가치를 하나의 스토리로 보여주는 순서다.

  1. [0:00] 관제 화면 오픈 — 상단 티커가 흐르고 서비스 LED가 순차 점등(부팅 시퀀스), 상태바 "SYSTEM ONLINE". "이건 실시간으로 도는 관제 플랫폼입니다."
  2. [0:10] 스캔 입력 — 스캔 콘솔에 shinhan-otp.xyz 입력. ~100ms 내 danger 73 + 근거 카드 3개(brand_impersonation·suspicious_tld·phishing_keywords)가 뜬다. "AI가 아니라 규칙이 왜 위험한지 설명합니다."
  3. [0:25] 킬샷 — 그래프가 뻗는다 — 판정과 동시에 관계망 그래프에 입력 노드가 추가되고, 공유 IP 203.0.113.44를 통해 기존 캠페인('은행피싱-B')으로 엣지가 뻗어 나간다. 카메라가 새 노드로 포커스. "흩어진 도메인이 같은 조직임을 드러냅니다."
  4. [0:45] 실시간·다중채널 — 우측 라이브 피드에 신고/스캔 이벤트가 4초마다 흐른다. deck.gl 지도에 위협 좌표 표시. "브라우저 확장과 모바일이 같은 판정을 공유합니다."

법률 · 개인정보 · 악용 방지

ScamGraph는 전화·계좌·도메인을 위험도와 함께 다룬다. 그런데 이 식별자의 소유자가 곧 범죄자는 아니다(발신번호 조작·대포통장 명의도용·도메인 침해). 따라서 표현·개인정보·악용방지가 제품의 필수 방어선이다. 문서: docs/{legal-privacy,expression-guidelines,abuse-defense,dpia-checklist}.md.

면책 관련 문서는 엔지니어링 내부 설계 기준이며 법률 자문이 아니다. 출시 전 반드시 한국 변호사·개인정보 전문가의 실제 법률검토(형법 명예훼손·업무방해, PIPA, 정보통신망법, 신용정보법, 통신비밀보호법, 전기통신사업법, 위치정보법)를 받아야 한다. 검토 전 실서비스 공개 배포 금지. 해커톤 데모는 시드/폴백 데이터로 운영해 실제 개인 식별자 노출을 피한다.

심사위원 예상 질문 & 답변

Q. "AI가 아니라면서 어떻게 정확한가요?"

A. 24개 이상의 명시적 신호(브랜드 사칭, Levenshtein 타이포스쿼팅, UTS#39 혼동문자, 퓨니코드, 인코딩 IP, 위험 TLD, 피싱 키워드, URL 구조)를 가중 합산합니다. 정상 도메인 화이트리스트로 오탐을 억제하고, 라벨링된 표본으로 평가 하네스가 정확도를 측정합니다(→ §7). AI가 아니라서 왜 그 점수인지 근거를 항상 재현할 수 있는 게 오히려 강점입니다.

Q. "오탐이 나면 어떻게 하나요?"

A. 세 단계입니다. ① 사전: 정상 도메인 화이트리스트로 즉시 safe. ② 사후: 누구나 접근 가능한 이의제기 폼. ③ 회수: 확정 시 긴급 정정 delta로 확장·모바일에서 즉시 rule 제거 + tombstone + 색인 purge. 침해된 정상 사이트는 루트 전체를 악성으로 단정하지 않습니다.

Q. "무고한 사람을 사기범으로 몰아 명예훼손 아닌가요?"

A. 발신번호는 조작 가능하고 대포통장은 명의도용이며 도메인은 침해될 수 있어 소유자 ≠ 범죄자입니다. 그래서 "사기범" 단정을 금지하고 "관측된/신고된/정황이 있는" 프레임만 씁니다. 식별자는 마스킹·defang하고 계좌 명의자는 수집하지 않으며 상세 페이지는 noindex입니다.

Q. "개인정보를 침해하지 않나요?"

A. "수집하지 않은 데이터는 유출될 수 없다"가 원칙입니다. SMS 원문·통화 내용은 기기 밖으로 나가지 않고(URL/지표만), 방문기록은 서버에 안 보내며(확장은 로컬 blocklist로 판정), 전화/계좌는 hash lookup·마스킹만 합니다.

Q. "네트워크가 끊기면 데모가 망하지 않나요?"

A. 3중 폴백입니다. 스캔은 무네트워크 quick_assess, 그래프는 SeedGraph 폴백, 실시간 피드는 4초 하트비트, 통계/지도는 시드. 외부 의존이 전부 죽어도 화면이 살아 있습니다.

Q. "경쟁사가 대량 신고로 정상 사이트를 차단시킬 수 있나요?"

A. ReportDefense가 막습니다. 신고자 익명 해시로 dedup(한 명이 반복해도 1), 시간창 burst 제한, 독립 신고자 수(DISTINCT)로만 집계, 정상 도메인 신고는 검토 큐로만. 신고 1건이 단독으로 danger/blocklist가 되는 일은 없고 ⑤단계는 운영자 dual-approval 필수입니다.

Q. "학생 프로젝트치고 규모가 큰데 실제로 도나요?"

A. 네. docker compose up 하나로 10개 서비스가 뜨고, 자동화 테스트와 k6 부하 테스트(→ §7)가 있으며, 브라우저 확장과 모바일 앱까지 같은 판정을 공유합니다. 모든 킬샷은 시드/폴백으로 라이브에서 항상 동작합니다.

Q. "블록리스트는 어떻게 배포하나요?"

A. 게이트웨이가 정규화된 지표를 SHA-256 해시·버전과 함께 스냅샷/델타로 제공하고, 확장은 이를 declarativeNetRequest 로컬 규칙으로 변환해 브라우저가 방문기록을 서버에 보내지 않고 즉시 차단합니다. 모바일도 오프라인 blocklist로 같은 방식으로 동작합니다.
