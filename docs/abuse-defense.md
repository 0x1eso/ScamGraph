# ScamGraph 악용 방지 정책 (Abuse Defense)

> 신고 플라이휠의 무결성 보호 · 신고 승격 정책 · 이의제기/정정 절차 · 서비스 역이용 방어.
> 출처: `roadmap.md` §7.3~7.5 를 실사용 가능한 정책으로 확장. 짝 문서: [`expression-guidelines.md`](expression-guidelines.md), [`legal-privacy.md`](legal-privacy.md).

## 0. 위협 모델

ScamGraph 의 커뮤니티 신고 플라이휠은 강력하지만, 동시에 **공격 표면**이다. 방어 대상:

| 공격 | 목표 | 대응 절 |
|---|---|---|
| **신고 poisoning** | 무고한 번호/계좌/도메인을 대량 허위신고로 차단시켜 업무방해·괴롭힘 | §1~§3 |
| **allowlist 우회 / 경쟁사 공격** | 정상 사업자·경쟁사를 위험으로 오도 | §3 승격정책, §4 이의제기 |
| **서비스 역이용** | 미탐 URL 테스트·차단기준 역공학·수사연결 파악·데이터 수집 | §5 |
| **인프라 공격** | 크롤러 SSRF·대량 크롤로 시스템 악용 | §5.2 |

기본 태도: **1건의 신고는 신호일 뿐, 판정이 아니다.** 판정은 다층 검증을 거친 승격(§3)을 통해서만 상향된다.

## 1. 신고 poisoning 방어 계층

`POST /report` 로 유입되는 모든 신고에 아래 계층을 순서대로 적용한다.

| 계층 | 통제 | 세부 |
|---|---|---|
| **L1 유입 제어** | rate limit · CAPTCHA/인증 | 계정/기기/IP 별 한도, 신규계정 영향력 제한, 봇 차단 |
| **L2 중복 제거** | dedup | 동일 증거·동일 대상 반복 신고를 1로 집계(동일 문구·타임스탬프 근접·동일 출처) |
| **L3 독립성 평가** | 신고자 독립성 | 서로 다른 계정이 담합(동일 IP 대역·동일 디바이스 지문·동시 burst)인지 판별 → 담합은 가중치 감산 |
| **L4 burst 탐지** | 이상 급증 | 특정 대상에 대한 단기 신고 급증을 anomaly detection 으로 플래그 → 자동 승격 보류·moderation queue |
| **L5 신고자 reputation** | 과거 적중률 | 신고자별 과거 신고의 확정률로 가중치 부여(Beta 사전분포), 저신뢰 신고자 영향 축소 |
| **L6 출처 분리** | 검증출처 vs 일반신고 | 경찰/은행/신뢰 feed = **검증출처**, 시민신고 = **일반신고**. 두 채널을 분리 집계·표기(expression §3) |

**신고 신뢰도 산정**
- 시민 신고 수는 **원시 카운트가 아니라** Wilson score / Beta-Binomial 신뢰구간 + **시간감쇠**로 환산(roadmap §1.8).
- 담합·burst·저신뢰 신고자 가중치는 최종 신뢰도에서 감산.

## 2. 검증 출처 vs 일반 신고 분리

| 구분 | 출처 예 | 판정 반영 | UI 표기 |
|---|---|---|---|
| **검증 출처(confirmed)** | 경찰청·금융기관·신뢰 위협 feed(OpenPhish/URLhaus/ThreatFox) | 단독으로 "확인된 위협" 승격 가능 | 실선 "확인됨" 배지 |
| **일반 신고(reported)** | 시민·사용자 신고 | 단독 승격 불가 → 표시·후보까지만 | 물음표 "N명 신고" 배지 |
| **기술 신호(inferred/confirmed)** | 규칙엔진·그래프 상관 | 강신호는 신고와 결합해 승격 | 점선/실선 배지 |

## 3. 신고 승격 정책 (핵심)

판정 등급(`grade`: `safe|caution|warning|danger`)과 blocklist 반영은 **아래 표의 조건을 충족할 때만** 상향한다. 신고 1건으로 danger·blocklist 가 되는 일은 없다.

### 3.1 승격 정책 표

| 단계 | 트리거 조건 | 결과(grade/조치) | 노출 | 승인 |
|---|---|---|---|---|
| **① 저장만** | 신고 1건 | 판정 변경 없음, 내부 저장만 | 미노출(또는 "신고 접수") | 자동 |
| **② 다수 신고 표시** | 독립 신고자 다수(dedup·독립성 통과) | `caution` 신호 가산, "다수 신고" 표시 | "N명 신고" 배지 | 자동 |
| **③ 위험 후보** | 신고 + **강한 기술신호**(feed 근접·homoglyph·form 신호 등) 결합 | `warning` 후보 | "위험 후보" · 근거 노출 | 자동 + 사후 검토 |
| **④ 확인된 위협** | **신뢰 feed 확인** 또는 검증출처(경찰/은행) 일치 | `danger` | "확인된 위협" · 근거 노출 | 자동(검증출처) |
| **⑤ 전체 blocklist** | **운영자 검토 + 복수 강증거** | signed blocklist 반영(확장/모바일 배포) | 차단 · interstitial | **운영자 dual approval 필수** |

### 3.2 승격 가드레일

- **allowlist 대상 신고**는 즉시 차단하지 않고 **검토 큐로**(정상 사업자 보호, roadmap §7.3).
- **high-impact 판정(④⑤)** 은 **dual approval**(운영자 2인 또는 검증출처+운영자) 필수.
- **서브도메인/침해 도메인**은 루트 도메인 전체를 blocklist 에 넣지 않는다(roadmap §1.5, R2 in legal-privacy).
- **공용 호스팅 IP**는 전체 차단 금지(개별 도메인 단위 차단, roadmap §5.1).
- 모든 ④⑤ 승격은 **판정ID·근거·승인자**를 audit log 에 남긴다.
- blocklist rule 은 **allowlist 가 우선**하도록 priority 설정(roadmap §5.1).

### 3.3 moderation queue

- burst/allowlist/dual-approval 대상은 운영자 moderation queue 에 적재.
- 큐 항목은 근거(reasons)·신고 출처·신뢰도·이의제기 여부와 함께 표시.
- 처리 결과(승격/기각/보류)는 audit log + 투명성 보고서 통계에 반영.

## 4. 이의제기 · 정정 절차

누구나(비로그인 포함) 접근 가능한 **appeal form** 을 제공한다. 잘못된 공개는 되돌릴 수 있어야 한다(roadmap §7.4).

### 4.1 처리 흐름

```
접수(appeal form) → 소유권/근거 확인 → (피해 크면)임시 suppress → 심사 → 판정변경/기각 → 정정 delta 배포 → tombstone 기록
```

### 4.2 SLA

| 단계 | 목표 시간(제안) | 조치 |
|---|---|---|
| 접수 확인 | 즉시(자동) | 접수번호 발급, "이의제기 검토 중" 배지 표시 |
| 임시 조치 | 24시간 이내 | 공개 피해가 크면 **임시 suppress**(노출 중단) |
| 최종 판정 | 영업일 기준 N일(법률검토 시 확정) | 변경/기각 사유 기록 |
| 정정 전파 | 확정 즉시 | 긴급 **정정 delta** → 모바일/확장 회수, 색인/캐시 purge |

### 4.3 정정 메커니즘

- **임시 suppress** — 심사 중 공개 노출 중단(표현 §7 "이의제기 검토 중").
- **tombstone** — blocklist·판정에서 제거 시 "정정됨" 표식 유지(재유입 방지·감사).
- **긴급 정정 delta** — signed feed delta 로 확장/모바일에서 즉시 rule 제거(roadmap §5.1, §7.4).
- **캐시/색인 purge** — CDN·검색색인에서 제거.
- **파트너 correction webhook** — 파트너 소비자에게 정정 이벤트 전파(roadmap §2.3).
- **신고자 개인정보 미공개** — 이의제기 처리 과정에서도 신고자 신원 비공개.

### 4.4 소유권 증명

- 도메인: DNS TXT verification · 관리 이메일.
- 전화/계좌: 본인확인 절차(법률검토로 확정). 명의도용 피해자 보호 경로 별도.
- 증명 없이도 **오탐 신고(제보)** 채널은 열어 두되, 판정 변경은 근거 검토 후.

## 5. 서비스 역이용 방어 & 투명성

공격자가 ScamGraph 자체를 도구로 악용하는 것을 막는다(roadmap §7.5).

### 5.1 정보 노출 통제

| 역이용 시나리오 | 방어 |
|---|---|
| **미탐 URL 테스트**(공격자가 자기 URL이 탐지되는지 확인) | 익명 결과는 **요약만**, 내부 임계값·규칙 세부 비공개. anomaly detection 으로 반복 테스트 탐지 |
| **차단기준 역공학** | 세부 근거·점수 산식·규칙버전 전체는 인증 조사자만. 공개는 등급+대표 근거 |
| **수사연결 파악**(어떤 캠페인이 수사 중인지 탐색) | 수사 중 비공개 사건 분리, 조사자 권한 분리(internal/partner/public evidence level) |
| **대량 데이터 수집** | bulk/batch API **승인제**, rate limit, reverse lookup 금지(expression §6) |
| **경쟁사 대량 신고** | §1 poisoning 방어 + allowlist 검토 큐(§3.2) |

### 5.2 크롤러/인프라 방어

- **SSRF 방어** — private IP·클라우드 metadata endpoint 차단, DNS rebinding 방어, egress allowlist, 응답 크기·시간 상한, 크롤러 격리망(roadmap §1.10, §6).
- **크롤 도메인별 동시성 제한** — 대상 서버에 대한 남용 방지.
- **격리 실행** — 크롤러는 분리된 네트워크에서 실행.

### 5.3 증거 레벨 분리

| 레벨 | 접근 | 내용 |
|---|---|---|
| **public** | 익명 | 등급 요약 + 대표 근거(마스킹) |
| **partner** | 인증 파트너 | 구조화 지표·delta·campaign 코드 |
| **internal** | 조사자/운영자 | 전체 식별자·상세 그래프·수사 중 사건 |

### 5.4 투명성 연계

정기 투명성 보고서로 신뢰를 확보한다(양식: [`transparency-report-template.md`](transparency-report-template.md)).

- 판정 정책·승격 기준(요지) 공개
- feed 출처·라이선스
- 삭제/정정 건수, 이의제기 인용률
- (가능 범위) 법집행 요청 통계

## 6. 운영 체크리스트

- [ ] `POST /report` 에 rate limit·dedup·독립성·burst·reputation 계층이 모두 적용되는가?
- [ ] 신고 1건이 단독으로 `danger`/blocklist 를 만들 수 없는가?(승격 표 준수)
- [ ] ④⑤ 승격에 dual approval + audit log 가 강제되는가?
- [ ] allowlist 대상 신고가 검토 큐로 가는가?
- [ ] 침해 도메인·공용 IP 를 루트/전체 차단하지 않는가?
- [ ] 이의제기 form 이 비로그인에서도 접근 가능한가?
- [ ] 임시 suppress·tombstone·긴급 정정 delta 가 동작하는가?
- [ ] 익명 결과가 요약만 노출하고 내부 임계값을 감추는가?
- [ ] bulk API 가 승인제·rate limit 인가?
- [ ] 크롤러 SSRF/격리/동시성 제한이 적용되는가?
- [ ] 신고자 개인정보가 전 과정에서 비공개인가?
- [ ] false-report poisoning 시뮬 테스트가 CI 에 있는가?(roadmap §6 테스트)
